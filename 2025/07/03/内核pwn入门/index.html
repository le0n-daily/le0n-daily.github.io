关于64位128TB 用户空间 + 128TB 内核空间

您提到的数字非常具体，这涉及到 x86-64 架构的一个特定设计。

  • 64位的潜力：一个 64 位的指针理论上可以寻址 2^64 字节，这大约是 16 EB (Exabytes)，一个天文数字。
  • 现实的限制 (Canonical Addresses)：目前的 x86-64 架构 CPU 并没有使用全部 64 位来寻址，而是通常只使用了低 48 位。这要求一个有效的地址，其第 47 位到第 63 位必须全部相同（要么全是 0，要么全是 1）。这种规范格式的地址被称为规范地址 (Canonical Address)。
    • 这 48 位地址空间提供了 2^48 = 256 TB 的虚拟地址空间。
  • 内核与用户的划分：Linux 内核非常巧妙地利用了这个“规范地址”的特点，将这 256TB 的虚拟地址空间一分为二：
    • 用户空间 (User Space)：地址范围从 0x00000000_00000000 到 0x00007FFF_FFFFFFFF。这些地址的最高有效位都是 0。这部分空间的大小正好是 128 TB。这部分是每个进程私有的。
    • 内核空间 (Kernel Space)：地址范围从 0xFFFF8000_00000000 到 0xFFFFFFFF_FFFFFFFF。这些地址的最高有效位都是 1。这部分的大小也是 128 TB。这部分空间被所有进程共享，但只有在内核态才能访问。

关于32位的3GB 用户空间 + 1GB 内核空间。

根本限制：32 位的寻址能力

  • 一个 32 位的指针，其所能表示的不同地址数量是 2^32。
  • 2^32 字节 = 4,294,967,296 字节 = 4 GB。
  • 这意味着，在 32 位架构上，任何一个进程的虚拟地址空间总大小被死死地限制在了 4GB。

对于系统中的每一个进程，它的 4GB 虚拟地址空间都被划分成如下两部分：

  1. 用户空间 (User Space): 3GB
    • 地址范围: 0x00000000 到 0xBFFFFFFF。
    • 内容: 这里存放着进程私有的一切，包括：
      • 程序代码（.text段）
      • 已初始化的数据（.data段）
      • 未初始化的数据（.bss段）
      • 堆（Heap），用于动态内存分配（如 malloc）
      • 栈（Stack），用于函数调用和局部变量
      • 内存映射的库（如 libc.so）和文件。
    • 属性: 这部分空间是进程私有的。进程 A 的 3GB 空间和进程 B 的 3GB 空间通过页表映射到了不同的物理内存上，它们互相隔离。
  2. 内核空间 (Kernel Space): 1GB
    • 地址范围: 0xC0000000 到 0xFFFFFFFF。
    • 内容: 这里存放着操作系统的核心代码和数据。
    • 属性: 这部分空间是所有进程共享的。无论当前哪个进程在运行，它的虚拟地址空间的高 1GB 都映射到同一块物理内存，即内核所在的物理内存。进程在用户态时没有权限访问这部分地址，只有当通过系统调用进入内核态后，CPU 的权限级别提升，才能访问这部分内存。

关于mm_struct

mm_struct 结构体是 Linux 内核中描述一个进程虚拟地址空间的核心数据结构，可以说是进程内存管理的“总司令部”。它包含了管理进程内存所需的所有信息。

这个结构体非常庞大且复杂，其定义位于内核源码的 include/linux/mm_types.h 文件中。由于内核版本更迭，字段会略有增删，但其核心成员和设计思想是稳定的。

以下是 mm_struct 的一些关键和有代表性的字段，并按照功能进行分类和注释，帮助您更好地理解。

总结：

数据结构 在 mm_struct 中的字段 擅长操作 主要应用场景 时间复杂度
VMA 链表 mmap 顺序遍历 (Iteration) fork(), /proc/pid/maps, 跨多个VMA的munmap 遍历: O(n)
VMA 红黑树 mm_rb 快速查找 (Lookup) 缺页中断 (Page Fault), mmap时检查地址冲突 查找/插入/删除: O(log n)

mm_struct 就像一个进程的内存蓝图，它通过以下几个核心组件来工作：

  1. VMA 链表和红黑树 (mmap, mm_rb)：这是灵魂所在。它将进程的虚拟地址空间划分为一个个具有不同属性（如可读、可写、可执行）的 vm_area_struct 区域。当你调用 mmap 时，内核就是在这里为你创建一个新的 VMA。
  2. 页全局目录 (pgd)：这是通往硬件翻译的桥梁。pgd 指向该进程页表的顶层结构，CPU 的 MMU 正是利用它来将虚拟地址翻译为物理地址。
  3. 内存统计 (total_vm, rss_stat)：提供了对进程内存使用情况的宏观掌控，也是 top、free 等命令的数据来源之一。
  4. 锁 (mmap_lock)：保证了在多线程环境下，对内存布局的修改是安全的。

每个用户进程都有其自己唯一的 mm_struct。而内核线程则比较特殊，它们没有自己的地址空间，所以它们的 task_struct->mm 字段通常是 NULL，它们运行时会借用上一个用户进程的 mm_struct。

关于中断

Linux内核通过中断技术，响应硬件发出的异步信号，暂停当前任务以立即处理紧急事件，从而实现高效的I/O操作和任务调度。

关于vmlloc、kmalloc、mmap

kmalloc、vmalloc 和 mmap 是 Linux 内核和驱动开发中至关重要的三个内存操作机制。它们在不同的场景下使用，理解其区别是编写健壮、高效内核代码的基础。

kmalloc / kfree：内核的主力内存分配方式

kmalloc 是内核中最常用、最高效的内存分配函数。

void *kmalloc(size_t size, gfp_t flags);

  • 作用: 分配一块物理上连续并且虚拟地址上也连续的内存。
  • 核心特点 (物理连续): 这是 kmalloc 最重要的特征。因为它分配的内存在物理 RAM 中是连成一片的，所以非常适合需要进行 DMA (Direct Memory Access) 操作的硬件设备。DMA 控制器通常不理解虚拟地址，只能操作物理地址。
  • 分配来源: kmalloc 的内存来自于内核的 slab/slub 分配器，这是一套为分配小块内存而高度优化的缓存机制。
  • 分配大小: 通常用于分配相对较小的、大小固定的内存块（一般不超过几兆字节，上限为 KMALLOC_MAX_SIZE）。
  • 速度: 非常快，因为大部分情况下是从预先分配好的缓存池中直接获取，几乎没有额外开销。
  • gfp_t flags: 这是一个关键参数，用于指定分配行为。
    • GFP_KERNEL: 最常用的标志。表示可以在需要时睡眠（阻塞），等待内存变为可用。因此，它不能在中断上下文或持有自旋锁时使用。
    • GFP_ATOMIC: 表示分配必须是原子的，绝不能睡眠。如果当前没有可用内存，它会立即返回 NULL。这是在中断处理程序等不能阻塞的代码中唯一可以使用的标志。

void kfree(const void *objp);

  • 作用: 释放由 kmalloc 分配的内存块，将其归还给 slab 分配器。
  • 规则: kfree 必须用于释放 kmalloc 分配的内存，不能混用。传入 NULL 是安全的操作。

一句话总结 kmalloc: 用于为设备驱动和内核核心功能分配快速、物理连续的小块内存。

vmalloc / vfree：为大块虚拟内存而生

vmalloc 用于分配大块的、对物理连续性没有要求的内存。

void *vmalloc(unsigned long size);

  • 作用: 分配一块虚拟地址上连续，但物理上不一定连续的内存。
  • 核心特点 (虚拟连续): 内核会将来自不同物理位置的零散内存页（Pages），通过修改内核的页表，将它们映射成一块看起来连续的虚拟地址空间。
  • 分配来源: 直接从内核的伙伴系统（Buddy System）中请求物理页，然后构建页表来映射它们。
  • 分配大小: 非常适合需要大块内存（例如几十兆或更大）的场景，比如为某些模块加载大的固件，或者创建大的缓冲区。
  • 速度: 比 kmalloc 慢得多。因为它涉及到查找空闲物理页、修改页表等一系列更复杂的操作。
  • 是否可睡眠: vmalloc 可能会睡眠以等待内存，因此不能在中断上下文中使用。

void vfree(const void *addr);

  • 作用: 释放由 vmalloc 分配的内存。它会解除页表映射，并将底层的物理页归还给伙伴系统。
  • 规则: 与 vfree 配对使用，不能混用。

一句话总结 vmalloc: 用于在内核中分配大块的、虚拟地址连续的内存，不关心物理上是否连续。

mmap / munmap：连接内核与用户的桥梁

mmap 本身是一个系统调用，它不是一个像 kmalloc 那样直接供内核其他部分使用的分配函数。它的主要职责是在用户进程的虚拟地址空间中创建一块新的内存映射。

从内核的角度看，响应 mmap 系统调用的过程是：

  • 作用: 在调用进程的地址空间中，创建一个新的 VMA (vm_area_struct)。这个 VMA 可以是：

    1. 文件映射 (File-backed): 将一个文件的某一部分映射到进程的地址空间。当用户访问这块内存时，内核通过缺页中断机制，按需从磁盘读取文件内容到物理内存，并建立页表映射。对这块内存的修改也可以被写回文件。
    2. 匿名映射 (Anonymous): 创建一块不与任何文件关联的内存区域，通常用作程序的堆或栈。这块内存会被初始化为零。

  • 核心特点: mmap 是实现用户空间与内核空间（特别是设备驱动）进行零拷贝 (Zero-copy) 数据交换的关键。驱动程序可以实现 mmap 接口，将设备的内存（如显存、硬件缓冲区）直接映射到用户空间，用户程序可以直接读写这块内存，避免了数据在内核和用户空间之间来回拷贝的开销。

  • 释放: 用户空间的程序通过调用 munmap 系统调用来解除这段映射，内核会回收对应的 VMA 和相关的物理内存页。

一句话总结 mmap: 它是用户程序向内核申请创建内存映射的机制，是实现高效 I/O 和进程内存管理的核心系统调用。

总结对比

特性 kmalloc vmalloc mmap (从内核视角)
分配目标 内核自身使用 内核自身使用 用户进程的地址空间
物理连续性 是 否 取决于底层（通常是否）
虚拟连续性 是 是 是
常用大小 小块 (< KMALLOC_MAX_SIZE) 大块 任意大小
性能/速度 非常快 较慢 开销较大（涉及VMA操作）
是否可睡眠 取决于flags (GFP_ATOMIC不可) 是 是（在系统调用上下文中）
主要用途 DMA、描述符、小缓冲区 大缓冲区、模块加载 驱动与用户空间数据交换、文件映射

关于task_struct

task_struct 是 Linux 内核中最核心、最庞大的数据结构，没有之一。它就是内核对一个进程或线程的完整描述，通常被称为**“进程描述符”(Process Descriptor)**。

理解了 task_struct，就等于拿到了理解内核如何管理进程的钥匙。

这个结构体定义在内核源码的 include/linux/sched.h 文件中。由于它极其庞大（在现代内核中有数百个字段），我们不可能逐一讲解。最好的方式是将其按功能模块进行分类解读，这能让你看清它的设计脉络。

task_struct 的核心作用：进程的“身份证”与“档案袋”

你可以把 task_struct 想象成一个人的全部档案，里面包含了：

  • 个人信息: 进程 ID (PID)、状态（在干嘛）、名字。
  • 工作安排: 调度策略、优先级、被分配到哪个 CPU 上运行。
  • 资产清单: 拥有哪些内存空间 (mm_struct)。
  • 社会关系: 父进程是谁、子进程有哪些。
  • 权限与凭证: 用户 ID、组 ID、拥有的能力 (Capabilities)。
  • 随身物品: 打开了哪些文件。

总结

task_struct 是内核数据结构的中心枢纽。它通过一系列的指针，将进程的状态、调度、内存、文件、凭证、信号等所有方面的信息全部组织和链接起来，形成了一个完整而复杂的管理体系。

对于 Pwn 学习者来说，理解以下几个指针指向的结构体尤为重要：

  • cred: 提权的核心目标。
  • mm: 理解进程虚拟内存布局，为用户态 Pwn 和内核利用打下基础。
  • files: 在一些利用场景中，可能需要操作文件描述符表（如重定向）。

ioctl

ioctl也是一个系统调用，用于与设备通信

int ioctl(int fd, unsigned long request, ...) 的第一个参数为打开设备 (open) 返回的文件描述符，第二个参数为用户程序对设备的控制命令，再后边的参数则是一些补充参数，与设备有关

内核被设计成可扩展的，可以加入一个称为设备驱动的模块，驱动的代码允许在内核空间运行而且可以对设备直接寻址。对设备驱动的请求是一个以设备和请求号码为参数的 ioctl 调用，如此内核就允许用户空间访问设备驱动进而访问设备而不需要了解具体的设备细节，同时也不需要一大堆针对不同设备的系统调用

Loadable Kernel Modules(LKMs)

可加载核心模块 (或直接称为内核模块) ,是内核空间的可执行程序，包括驱动程序（设备驱动、文件系统驱动）和内核扩展模块,和用户态的可执行程序相同，Linux 下为 ELF，Windows 下为 exe/dll，mac 下为 MACH-O

模块可以被单独编译，但不能单独运行，运行时被链接到内核作为内核的一部分在内核空间运行，这与运行在用户控件的进程不同

Linux 内核之所以提供模块机制，是因为它本身是一个单内核 (monolithic kernel)。单内核的优点是效率高，因为所有的内容都集合在一起，但缺点是可扩展性和可维护性相对较差，模块机制就是为了弥补这一缺陷。

相关指令

  • rmmod: 从内核中卸载指定模块
  • lsmod: 列出已经加载的模块
  • modprobe: 添加或删除模块，加载模块时会查找依赖关系
