关于64位128TB 用户空间 + 128TB 内核空间

您提到的数字非常具体，这涉及到 x86-64 架构的一个特定设计。

  • 64位的潜力：一个 64 位的指针理论上可以寻址 2^64 字节，这大约是 16 EB (Exabytes)，一个天文数字。
  • 现实的限制 (Canonical Addresses)：目前的 x86-64 架构 CPU 并没有使用全部 64 位来寻址，而是通常只使用了低 48 位。这要求一个有效的地址，其第 47 位到第 63 位必须全部相同（要么全是 0，要么全是 1）。这种规范格式的地址被称为规范地址 (Canonical Address)。
    • 这 48 位地址空间提供了 2^48 = 256 TB 的虚拟地址空间。
  • 内核与用户的划分：Linux 内核非常巧妙地利用了这个“规范地址”的特点，将这 256TB 的虚拟地址空间一分为二：
    • 用户空间 (User Space)：地址范围从 0x00000000_00000000 到 0x00007FFF_FFFFFFFF。这些地址的最高有效位都是 0。这部分空间的大小正好是 128 TB。这部分是每个进程私有的。
    • 内核空间 (Kernel Space)：地址范围从 0xFFFF8000_00000000 到 0xFFFFFFFF_FFFFFFFF。这些地址的最高有效位都是 1。这部分的大小也是 128 TB。这部分空间被所有进程共享，但只有在内核态才能访问。

关于32位的3GB 用户空间 + 1GB 内核空间。

根本限制：32 位的寻址能力

  • 一个 32 位的指针，其所能表示的不同地址数量是 2^32。
  • 2^32 字节 = 4,294,967,296 字节 = 4 GB。
  • 这意味着，在 32 位架构上，任何一个进程的虚拟地址空间总大小被死死地限制在了 4GB。

对于系统中的每一个进程，它的 4GB 虚拟地址空间都被划分成如下两部分：

  1. 用户空间 (User Space): 3GB
    • 地址范围: 0x00000000 到 0xBFFFFFFF。
    • 内容: 这里存放着进程私有的一切，包括：
      • 程序代码（.text段）
      • 已初始化的数据（.data段）
      • 未初始化的数据（.bss段）
      • 堆（Heap），用于动态内存分配（如 malloc）
      • 栈（Stack），用于函数调用和局部变量
      • 内存映射的库（如 libc.so）和文件。
    • 属性: 这部分空间是进程私有的。进程 A 的 3GB 空间和进程 B 的 3GB 空间通过页表映射到了不同的物理内存上，它们互相隔离。
  2. 内核空间 (Kernel Space): 1GB
    • 地址范围: 0xC0000000 到 0xFFFFFFFF。
    • 内容: 这里存放着操作系统的核心代码和数据。
    • 属性: 这部分空间是所有进程共享的。无论当前哪个进程在运行，它的虚拟地址空间的高 1GB 都映射到同一块物理内存，即内核所在的物理内存。进程在用户态时没有权限访问这部分地址，只有当通过系统调用进入内核态后，CPU 的权限级别提升，才能访问这部分内存。

关于mm_struct

mm_struct 结构体是 Linux 内核中描述一个进程虚拟地址空间的核心数据结构，可以说是进程内存管理的“总司令部”。它包含了管理进程内存所需的所有信息。

这个结构体非常庞大且复杂，其定义位于内核源码的 include/linux/mm_types.h 文件中。由于内核版本更迭，字段会略有增删，但其核心成员和设计思想是稳定的。

以下是 mm_struct 的一些关键和有代表性的字段，并按照功能进行分类和注释，帮助您更好地理解。

总结：

数据结构 在 mm_struct 中的字段 擅长操作 主要应用场景 时间复杂度
VMA 链表 mmap 顺序遍历 (Iteration) fork(), /proc/pid/maps, 跨多个VMA的munmap 遍历: O(n)
VMA 红黑树 mm_rb 快速查找 (Lookup) 缺页中断 (Page Fault), mmap时检查地址冲突 查找/插入/删除: O(log n)

mm_struct 就像一个进程的内存蓝图，它通过以下几个核心组件来工作：

  1. VMA 链表和红黑树 (mmap, mm_rb)：这是灵魂所在。它将进程的虚拟地址空间划分为一个个具有不同属性（如可读、可写、可执行）的 vm_area_struct 区域。当你调用 mmap 时，内核就是在这里为你创建一个新的 VMA。
  2. 页全局目录 (pgd)：这是通往硬件翻译的桥梁。pgd 指向该进程页表的顶层结构，CPU 的 MMU 正是利用它来将虚拟地址翻译为物理地址。
  3. 内存统计 (total_vm, rss_stat)：提供了对进程内存使用情况的宏观掌控，也是 top、free 等命令的数据来源之一。
  4. 锁 (mmap_lock)：保证了在多线程环境下，对内存布局的修改是安全的。

每个用户进程都有其自己唯一的 mm_struct。而内核线程则比较特殊，它们没有自己的地址空间，所以它们的 task_struct->mm 字段通常是 NULL，它们运行时会借用上一个用户进程的 mm_struct。

关于中断

Linux内核通过中断技术，响应硬件发出的异步信号，暂停当前任务以立即处理紧急事件，从而实现高效的I/O操作和任务调度。

关于vmlloc、kmalloc、mmap

kmalloc、vmalloc 和 mmap 是 Linux 内核和驱动开发中至关重要的三个内存操作机制。它们在不同的场景下使用，理解其区别是编写健壮、高效内核代码的基础。

kmalloc / kfree：内核的主力内存分配方式

kmalloc 是内核中最常用、最高效的内存分配函数。

void *kmalloc(size_t size, gfp_t flags);

  • 作用: 分配一块物理上连续并且虚拟地址上也连续的内存。
  • 核心特点 (物理连续): 这是 kmalloc 最重要的特征。因为它分配的内存在物理 RAM 中是连成一片的，所以非常适合需要进行 DMA (Direct Memory Access) 操作的硬件设备。DMA 控制器通常不理解虚拟地址，只能操作物理地址。
  • 分配来源: kmalloc 的内存来自于内核的 slab/slub 分配器，这是一套为分配小块内存而高度优化的缓存机制。
  • 分配大小: 通常用于分配相对较小的、大小固定的内存块（一般不超过几兆字节，上限为 KMALLOC_MAX_SIZE）。
  • 速度: 非常快，因为大部分情况下是从预先分配好的缓存池中直接获取，几乎没有额外开销。
  • gfp_t flags: 这是一个关键参数，用于指定分配行为。
    • GFP_KERNEL: 最常用的标志。表示可以在需要时睡眠（阻塞），等待内存变为可用。因此，它不能在中断上下文或持有自旋锁时使用。
    • GFP_ATOMIC: 表示分配必须是原子的，绝不能睡眠。如果当前没有可用内存，它会立即返回 NULL。这是在中断处理程序等不能阻塞的代码中唯一可以使用的标志。

void kfree(const void *objp);

  • 作用: 释放由 kmalloc 分配的内存块，将其归还给 slab 分配器。
  • 规则: kfree 必须用于释放 kmalloc 分配的内存，不能混用。传入 NULL 是安全的操作。

一句话总结 kmalloc: 用于为设备驱动和内核核心功能分配快速、物理连续的小块内存。

vmalloc / vfree：为大块虚拟内存而生

vmalloc 用于分配大块的、对物理连续性没有要求的内存。

void *vmalloc(unsigned long size);

  • 作用: 分配一块虚拟地址上连续，但物理上不一定连续的内存。
  • 核心特点 (虚拟连续): 内核会将来自不同物理位置的零散内存页（Pages），通过修改内核的页表，将它们映射成一块看起来连续的虚拟地址空间。
  • 分配来源: 直接从内核的伙伴系统（Buddy System）中请求物理页，然后构建页表来映射它们。
  • 分配大小: 非常适合需要大块内存（例如几十兆或更大）的场景，比如为某些模块加载大的固件，或者创建大的缓冲区。
  • 速度: 比 kmalloc 慢得多。因为它涉及到查找空闲物理页、修改页表等一系列更复杂的操作。
  • 是否可睡眠: vmalloc 可能会睡眠以等待内存，因此不能在中断上下文中使用。

void vfree(const void *addr);

  • 作用: 释放由 vmalloc 分配的内存。它会解除页表映射，并将底层的物理页归还给伙伴系统。
  • 规则: 与 vfree 配对使用，不能混用。

一句话总结 vmalloc: 用于在内核中分配大块的、虚拟地址连续的内存，不关心物理上是否连续。

mmap / munmap：连接内核与用户的桥梁

mmap 本身是一个系统调用，它不是一个像 kmalloc 那样直接供内核其他部分使用的分配函数。它的主要职责是在用户进程的虚拟地址空间中创建一块新的内存映射。

从内核的角度看，响应 mmap 系统调用的过程是：

  • 作用: 在调用进程的地址空间中，创建一个新的 VMA (vm_area_struct)。这个 VMA 可以是：

    1. 文件映射 (File-backed): 将一个文件的某一部分映射到进程的地址空间。当用户访问这块内存时，内核通过缺页中断机制，按需从磁盘读取文件内容到物理内存，并建立页表映射。对这块内存的修改也可以被写回文件。
    2. 匿名映射 (Anonymous): 创建一块不与任何文件关联的内存区域，通常用作程序的堆或栈。这块内存会被初始化为零。

  • 核心特点: mmap 是实现用户空间与内核空间（特别是设备驱动）进行零拷贝 (Zero-copy) 数据交换的关键。驱动程序可以实现 mmap 接口，将设备的内存（如显存、硬件缓冲区）直接映射到用户空间，用户程序可以直接读写这块内存，避免了数据在内核和用户空间之间来回拷贝的开销。

  • 释放: 用户空间的程序通过调用 munmap 系统调用来解除这段映射，内核会回收对应的 VMA 和相关的物理内存页。

一句话总结 mmap: 它是用户程序向内核申请创建内存映射的机制，是实现高效 I/O 和进程内存管理的核心系统调用。

总结对比

关于task_struct

task_struct 是 Linux 内核中最核心、最庞大的数据结构，没有之一。它就是内核对一个进程或线程的完整描述，通常被称为**“进程描述符”(Process Descriptor)**。

理解了 task_struct，就等于拿到了理解内核如何管理进程的钥匙。

这个结构体定义在内核源码的 include/linux/sched.h 文件中。由于它极其庞大（在现代内核中有数百个字段），我们不可能逐一讲解。最好的方式是将其按功能模块进行分类解读，这能让你看清它的设计脉络。

task_struct 的核心作用：进程的“身份证”与“档案袋”

你可以把 task_struct 想象成一个人的全部档案，里面包含了：

  • 个人信息: 进程 ID (PID)、状态（在干嘛）、名字。
  • 工作安排: 调度策略、优先级、被分配到哪个 CPU 上运行。
  • 资产清单: 拥有哪些内存空间 (mm_struct)。
  • 社会关系: 父进程是谁、子进程有哪些。
  • 权限与凭证: 用户 ID、组 ID、拥有的能力 (Capabilities)。
  • 随身物品: 打开了哪些文件。

总结

task_struct 是内核数据结构的中心枢纽。它通过一系列的指针，将进程的状态、调度、内存、文件、凭证、信号等所有方面的信息全部组织和链接起来，形成了一个完整而复杂的管理体系。

对于 Pwn 学习者来说，理解以下几个指针指向的结构体尤为重要：

  • cred: 提权的核心目标。
  • mm: 理解进程虚拟内存布局，为用户态 Pwn 和内核利用打下基础。
  • files: 在一些利用场景中，可能需要操作文件描述符表（如重定向）。

关于cred结构体

好的，这是一个非常核心的内核数据结构。struct cred 定义了进程执行操作时所拥有的所有凭证和权限。它的具体内容会随着内核版本的演进略有变化，这里提供一个基于较新内核版本（例如 5.x 或 6.x）的完整结构体定义，并对关键字段进行详细解释。

这个结构体定义通常位于内核源码的 include/linux/cred.h 文件中。

struct cred 完整定义

关键字段详细解析

  1. usage (atomic_t)

    • 作用：引用计数器。因为多个进程可能共享同一个 cred 结构体（例如，在 clone 系统调用时），这个字段记录了共享者的数量。当 usage 减到0时，这个结构体占用的内存才会被真正释放。这是防止UAF漏洞的重要机制。

  2. 用户和组ID (UIDs & GIDs)

    • uid / gid (真实ID)：标识进程的真正所有者。通常只有root进程（或拥有CAP_SETUID能力的进程）才能改变它们。
    • euid / egid (有效ID)：这是进行权限检查时最重要的ID。例如，当访问一个文件时，内核会比较进程的 euid 和 egid 与文件的所有者信息来决定访问权限。setuid 程序的核心就是改变 euid。
    • suid / sgid (保存的ID)：当一个进程通过 execve 执行一个 setuid 程序时，它原来的 euid 会被保存到 suid 中。这使得进程可以在需要时恢复回原来的权限。
    • fsuid / fsgid (文件系统ID)：专门用于文件系统相关的权限检查。在大多数情况下，它们与 euid/egid 相同，但在某些特殊场景（如NFS服务）下可能会不同。

  3. 能力 (Capabilities)

    • 在现代Linux系统中，传统的将所有特权赋予root用户的模式被打破了。能力 (Capabilities) 将root用户的特权分割成一组组离散的单元。一个进程可以只被授予它完成任务所必需的特定能力，而不是全部的root权限。
    • cap_permitted (许可集)：定义了该进程可能拥有的能力的上限。即使通过某种方式获得了某个能力，如果它不在许可集中，也无法生效。
    • cap_effective (有效集)：这是内核进行特权操作检查时实际使用的能力集。它必须是 cap_permitted 的一个子集。
    • cap_inheritable (继承集)：当执行 execve 时，这个集合中的能力可以被传递给新程序。
    • cap_bset (边界集)：系统范围内的能力上限，任何进程的能力都不能超过这个边界。
    • cap_ambient (环境集)：一种让非 setuid 程序也能继承能力的方式，解决了传统能力继承机制的一些问题。

  4. 安全模块 (Security Module)

    • security (void *)：这是一个非常重要的扩展点。它是一个指针，指向由当前系统启用的Linux安全模块（LSM），如 SELinux 或 AppArmor，所使用的私有数据。内核在进行权限检查时，除了检查传统的UID/GID和能力，还会调用LSM提供的钩子函数，通过这个 security 指针来获取额外的安全上下文，进行强制访问控制（MAC）。

  5. 密钥环 (Keyrings)

    • session_keyring, process_keyring 等：指向内核密钥管理服务的不同密钥环。用于安全地存储加密密钥、认证令牌等敏感信息。

在Pwn中的意义

对于内核漏洞利用（Pwn）来说，struct cred 是最终极的目标。只要能获得向内核任意地址写入数据的能力（Arbitrary Write），攻击者就会构造一个 cred 结构体的镜像，将其所有关键字段都设置为0或最高权限，然后用这个构造好的数据覆盖掉当前进程在内核中的 cred 结构体。

  • 将 uid, gid, euid, egid 等全部置为 0。
  • 将所有的 capability 集合（cap_permitted, cap_effective 等）全部置为 0xffffffffffffffff（即启用所有能力）。

一旦覆盖成功，当前进程就瞬间从一个普通用户进程变成了拥有所有权限的root进程，从而达到提权的目的。

ioctl

ioctl也是一个系统调用，用于与设备通信

int ioctl(int fd, unsigned long request, ...) 的第一个参数为打开设备 (open) 返回的文件描述符，第二个参数为用户程序对设备的控制命令，再后边的参数则是一些补充参数，与设备有关

内核被设计成可扩展的，可以加入一个称为设备驱动的模块，驱动的代码允许在内核空间运行而且可以对设备直接寻址。对设备驱动的请求是一个以设备和请求号码为参数的 ioctl 调用，如此内核就允许用户空间访问设备驱动进而访问设备而不需要了解具体的设备细节，同时也不需要一大堆针对不同设备的系统调用

Loadable Kernel Modules(LKMs)

可加载核心模块 (或直接称为内核模块) ,是内核空间的可执行程序，包括驱动程序（设备驱动、文件系统驱动）和内核扩展模块,和用户态的可执行程序相同，Linux 下为 ELF，Windows 下为 exe/dll，mac 下为 MACH-O

模块可以被单独编译，但不能单独运行，运行时被链接到内核作为内核的一部分在内核空间运行，这与运行在用户控件的进程不同

Linux 内核之所以提供模块机制，是因为它本身是一个单内核 (monolithic kernel)。单内核的优点是效率高，因为所有的内容都集合在一起，但缺点是可扩展性和可维护性相对较差，模块机制就是为了弥补这一缺陷。

  • modprobe: 添加或删除模块，加载模块时会查找依赖关系

改变自身

通过改变自身进程的权限，使其具有 root 权限,内核会通过进程的 task_struct 结构体中的 cred 指针来索引 cred 结构体，cred结构体成员中的 uid-fsgid 都为 0则具有 root 权限,这种提权方法分为定位和修改两个步骤，cred结构体在include/linux/cred.h

直接改cred结构体的内容
定位cred具体位置

  • 直接定位

    cred 结构体的最前面记录了各种 id 信息，对于一个普通的进程而言，uid-fsgid 都是执行进程的用户的身份,因此可以通过扫描内存来定位 cred

    **在实际定位的过程中，我们可能会发现很多满足要求的 cred，这主要是因为 cred 结构体可能会被拷贝、释放。**可以利用 usage 不为 0 来筛除掉一些 cred，但仍然会发现一些 usage 为 0 的 cred,这是因为 cred 从 usage 为 0， 到释放有一定的时间,此外，cred 是使用 rcu 延迟释放的

  • 间接定位

    • task_struct

      进程的 task_struct 结构体中会存放指向 cred 的指针，因此我们可以

      1. 定位当前进程 task_struct 结构体的地址
      2. 根据 cred 指针相对于 task_struct 结构体的偏移计算得出 cred 指针存储的地址
      3. 获取 cred 具体的地址

    • comm

      comm 用来标记可执行文件的名字，位于进程的 task_struct 结构体中, comm 在 cred 的正下方，所以可以先定位 comm ，然后定位 cred 的地址

      进程名字并不特殊的情况下可能会有多个同样的字符串，,可以使用 prctl 设置进程的comm为一个特殊的字符串，然后再开始定位 comm

  • 修改

    直接将 cred 中的 uid-fsgid 都修改为 0,当然修改的方式有很多种

    • 在我们具有任意地址读写后，可以直接修改 cred
    • 在我们可以 ROP 执行代码后，可以利用 ROP gadget 修改 cred
间接定位

不一定非得知道 cred 的具体位置，只需要能够修改 cred 即可

  1. 申请一块与 cred 结构体大小一样的堆块
  2. 释放该堆块

此种方法在较新版本内核中已不再可行，我们已无法直接分配到 cred_jar 中的 object，这是因为 cred_jar 在创建时设置了 SLAB_ACCOUNT 标记，在 CONFIG_MEMCG_KMEM=y 时（默认开启）cred_jar 不会再与相同大小的 kmalloc-192 进行合并

  • commit_creds(&init_cred)

    commit_creds() :将一个新的 cred 设为当前进程 task_struct 的 real_cred 与 cred 字段，调用该函数并传入一个具有 root 权限的 cred，则能直接完成对当前进程的提权工作

write()函数

exp的思路：

第一步：制造悬垂指针 (UAF)

  1. int fd = open(“/dev/test1”, 6);
    • 内核调用 test1_open，分配了一个32字节的缓冲区 kbuf_A。
    • 全局指针 test1_buffer 指向 kbuf_A。
  2. int fd2 = open(“/dev/test1”, 6);
    • 内核再次调用 test1_open，分配了一个32字节的缓冲区 kbuf_B。
    • 全局指针 test1_buffer 被覆盖，现在指向 kbuf_B。
  3. write(fd2, buf, 0x8a)————这里需要明白在执行write()时是怎么操作的
    • 内核调用 test1_write。此时 used=0, count=0x8a(138), 全局 length=32。
    • 0 + 138 > 32，条件成立，进入 else 分支。
    • dest = _kmalloc(138, …)：分配了一个138字节的新缓冲区 kbuf_C。
    • memcpy(dest, test1_buffer, 0)：把 kbuf_B 的内容（0字节）拷到 kbuf_C。
    • kfree(test1_buffer)：释放了32字节的 kbuf_B。
    • test1_buffer = dest：全局指针 test1_buffer 现在指向138字节的 kbuf_C。
    • ::length = 138：全局长度更新为138。
    • copy_from_user(…)：将用户数据写入 kbuf_C。
  4. close(fd2);
    • 内核调用 test1_release。
    • 函数执行 kfree(test1_buffer)。
    • 此时 test1_buffer 指向的是138字节的 kbuf_C。所以，kbuf_C 被释放了。
    • 关键点：test1_buffer 指针没有被清零，它现在成了一个指向刚刚被释放的138字节内存的悬垂指针 (Dangling Pointer)。

第二步：内存占位

  1. int pid = fork();
    • 内核为了创建子进程，需要分配一个 struct cred 结构体来存放其权限信息。
    • 内核的内存分配器（SLUB/SLAB）会寻找一块大小合适的空闲内存。
    • struct cred 的大小通常在120-170(这道题是128)字节之间，我们刚刚释放的138字节的内存块 (kbuf_C 所在的位置) 是一个完美的候选者！
    • “奇迹”发生：内核极大概率会在这块刚刚被释放的内存上分配新的 cred 结构体。
    • 结果：现在，悬垂指针 test1_buffer 实际上已经指向了子进程的 cred 结构体。

第三步：触发漏洞，完成写入

  1. 在子进程中执行 write(fd, buf, 0x28)
    • 内核再次调用 test1_write。此时 used 可能是0（被之前的read重置或新进程开始时为0），count=0x28(40)。全局 length 仍然是138。
    • 0 + 40 <= 138，条件成立，这次进入的是 if 分支！
    • 内核执行 copy_from_user(&test1_buffer[used], buf, v4);
    • 我们来分析此时 copy_from_user 的第一个参数（在x86-64架构下，它会被放入 rdi 寄存器）：
      • &test1_buffer[used] 等价于 test1_buffer + 0。
      • test1_buffer 是什么？是我们在第4步制造的、在第5步被 cred 结构体占据的悬垂指针。
      • 所以，rdi 寄存器中的地址，正是子进程 cred 结构体的起始地址！

再看一下主要函数（第一次做结合ai分析的）

babydriver_init 函数是这个内核模块（Kernel Module）的入口点。当你在终端中使用 insmod babydriver.ko 命令加载这个模块时，内核就会调用这个函数。

它的核心目的可以总结为一句话：创建一个名为 babydev 的字符设备（Character Device），并将其注册到系统中，以便用户空间的程序可以像读写文件一样与它进行交互。

全局状态

首先，请注意全局变量 babydev_struct。所有的函数都围绕着这个唯一的、共享的结构体进行操作。这意味着它的状态（分配的缓冲区指针和长度）在不同的系统调用（open, read, write, ioctl, close）之间是持续存在的，并且是全局共享的。

函数逐个分析（漏洞分析）

1. babyopen 和 babyrelease (UAF / 悬垂指针漏洞)

  • 它们的作用：

    • babyopen 在你执行 open("/dev/babydev", ...) 时被调用。它会分配一块 64 字节大小的内核内存，并将指针存放在 babydev_struct.device_buf 中。
    • babyrelease 在你执行 close() 时被调用，它会释放这块内存。

  • 漏洞：释放后重用 (Use-After-Free, UAF)

    • 请仔细看 babyrelease 函数。在它调用 kfree(babydev_struct.device_buf) 之后，它没有将指针 babydev_struct.device_buf 设置为 NULL。
    • babydev_struct.device_buf 现在包含了一个已经被释放的内存地址。这种指针被称为 “悬垂指针” (Dangling Pointer)。
    • 如果在下一次调用 open() 重新分配内存之前，任何其他函数（如 read 或 write）尝试使用这个悬垂指针，它将访问一块已经被内核回收的内存，从而导致 UAF 漏洞。这是一个非常严重的漏洞。

2. babywrite 和 babyioctl

  • 它们的作用：
    • babyioctl 是一个控制函数。如果你调用 ioctl(fd, 0x10001, new_size)，它会释放旧的缓冲区，并分配一个由你指定大小 (new_size) 的新缓冲区。
    • babywrite 应该从你的用户态程序拷贝数据到内核缓冲区。

3. babyread

利用思路（提权策略）

  1. 打开两次设备，通过ioctl将babydev_struct.device_buf大小变为的cred结构体的大小
  2. 释放第一个设备，fork出一个新的进程，这个进程的cred结构体就会放在babydev_struct.device_buf所指向的内存区域
  3. 使用第二个描述符，调用write向此时的babydev_struct.device_buf中写入28个0，刚好覆盖至uid和gid，实现root提权

写好exp过后，由于rootfs.cpio里并没有libc，所以编译的时候要使用静态编译，利用musl-gcc可以使文件占的内存更小。

内核态的 ROP 与用户态的 ROP 一般无二，只不过利用的 gadget 变成了内核中的 gadget，所需要构造执行的 ropchain 由 system("/bin/sh") 变为了 commit_creds(&init_cred) 或 commit_creds(prepare_kernel_cred(NULL))，当我们成功地在内核中执行这样的代码后，当前线程的 cred 结构体便变为 init 进程的 cred 的拷贝，我们也就获得了 root 权限，此时在用户态起一个 shell 便能获得 root shell。

分析

init_module()

proc_create 是 Linux 内核中的一个函数，用于创建一个新的 /proc 文件系统条目。这个函数常用于内核模块中，以便在 /proc 文件系统下创建一个新的文件，使得用户空间程序可以通过这个文件与内核模块进行交互。
这里创建一个名为 ： /proc/core

对于这里的fops,也只对core_write,core_ioctl,core_release进行了注册。

core_ioctl()

这里core_ioctl中定义了三种操作，分别是调用core_read()，设置全局变量off，调用core_copy_func()。

core_read()

这里的copy_to_user(),会把内核空间中的栈上的数据拷贝到a1,a1和off是我们可以控制的，因此可以利用这个函数来泄露canary

core_write()

core_write是将至多0x800个字节从指定缓冲区复制到name中去

core_copy_func()

漏洞整数溢出造成栈溢出，传入的参数a1是__int64有符号的整数(无符号：size_t 或 unsigned __int64)，与63是无符号整数，如果传入一个负数即可进入else分支实现栈溢出。

最后调用qmemcpy时转成了unsigned __int16。所以只需要将a1最低两个字节的数据随便设置成一个能装下name的长度，然后其余字节都是0xff就行了。我这里最后构造的a1是0xffffffffffff0100。

利用思路

  1. 通过调试设置off，利用core_read函数去读取canary
  2. 构造ROP链，用core_write函数往name中写入数据
  3. 调用core_copy_func，将name的内容写入栈上变量v1中，造成栈溢出，调用commit_creds(prepare_kernel_cred(0))提权。
